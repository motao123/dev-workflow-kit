# change-impact-analysis

用于分析某次改动会影响哪些模块、接口、配置、测试、文档和消费者。

## 适用场景

- 改一个点，但不知道会牵动哪里
- 改动可能跨模块、跨环境、跨消费者
- merge / release 风险取决于影响面是否摸清楚
- 想先知道“这次改动会波及谁”

## 标准流程

1. 确定改动点。
2. 找出直接影响面。
3. 推断间接影响面和回归风险区。
4. 区分必须验证和低风险区域。
5. 给出验证重点和 rollout 注意点。
6. 给出下一步建议。

## 推荐搭配

- 范围需要重切：`planning`
- 真正落地实现：进入当前项目的实现工作流
- 涉及发版形态：`release-rollout-strategy` / `ship-readiness`

## 调用示例

```text
用 change-impact-analysis 看一下这次改动会影响哪些模块、接口和验证面。
```
